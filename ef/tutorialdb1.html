Creating DB connection

The framework uses SqlDbConnection class to perform all connection-related operations.

To get an instance of the connection, the developer must use one of the database factory classes which are named as DbNameDbConnectionFactory (e.g) MssqlConnectionFactory and always has a method named Create which always has a following prototype:

SqlDbConnection Create(string connectionString)

or for asynchronous version

Task<SqlDbConnection> CreateAsync(string connectionString, CancellationToken?)

The delegate definitions are SqlDbConnectionFactory and SqlDbConnectionFactoryAsync

Use this method in order to get an instance of the connection.

Dispose the connection object when it is not in use.

You can also use UniversalSqlDbFactory to create a connection by the driver name.

DB-specific Implementations

Wherever the application needs to know any platform specific, an instance of SqlDbLanguageSpecifics may be obtained from the connection object via GetLanguageSpecifics() method.

Pooling

The framework is build over ADO.NET technology so it has a connection pooling embed. All connections that shares the same connection string aren't actually closed at disposition, but are returned back to the connection pool for further reuse.

So, it is recommended to limit lifetime of the connection object to quite a short period of time, e.g. one request handling, and dispose it as soon as it reasonably possible.

There is only once case when the behavior isn't clear and well-defined, it is in-memory SQLite database. In this case is it recommended to keep one instance of the connection during the whole lifecycle of the application.
